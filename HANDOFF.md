## What This Project Is

A **collector-grade vinyl pressing guide** for 50 essential jazz albums. Each guide contains:
- Pressing/reissue hierarchy (S tier down to F tier based on sound quality, build, and value)
- Detailed sound quality analysis per pressing, heavily sourced (Discogs reviews, YouTube shootouts, LondonJazzCollector, Steve Hoffman Forums, print magazines)
- Discogs links for each pressing, feeding a **live market-data pipeline** (see below)
- Buyer's guide with a **relative $–$$$$$ price-band scale** (not absolute dollar figures — see "Pricing conventions" below)
- Album metadata (MusicBrainz ID where available, cover art via a build-time resolution chain, release date)
- A "Related guides" cross-linking block at the bottom of every guide

**Not a:** streaming recommendation site, critical review list, or complete jazz discography. This is specifically about *vinyl pressings* for collectors who care about which version they're buying.

## Building and Deploying

```bash
cd jazzvinylguide
python3 build.py              # builds everything: index, all 50 albums, changelogs, status page, sitemap.xml
python3 build.py <slug>       # builds just one album (faster iteration)
```

This reads all 50 `.md` files, converts Markdown → HTML, generates `sitemap.xml`, resolves cover art (see below), and writes everything to the repo root / `albums/`. **No external API calls required for a basic build** — but a full build *does* call the GitHub API (for `dateModified`/changelog dates) and the iTunes Search API (for cover art fallback on albums without a working mbid). Both degrade gracefully if unreachable.

**Push to `main`** → go to the Netlify dashboard and trigger a manual deploy (autodeploy is intentionally off).

## Pricing Conventions (major change this session)

Two separate systems, don't confuse them:

1. **Tier table "Market" column** — live, sourced automatically from Discogs via a scheduled GitHub Actions workflow (`scripts/fetch_market_data.py`, weekly). Not something you write by hand.

2. **Buyer's Guide headers** — converted from absolute dollar figures (`**Under $50:**`) to a **relative 5-band scale**:
   - `$` = Budget (≤~$40) · `$$` = Affordable (~$40–90) · `$$$` = Mid-range (~$90–180) · `$$$$` = Premium (~$180–350) · `$$$$$` = Grail/Rare (~$350+)
   - Bands are absolute across the whole catalogue (not relative-within-guide), so `$$$` means roughly the same thing on every album's page.
   - The mechanical audit checks these are in non-decreasing order per guide.

3. **Prose price mentions** (the big cleanup this session) — every one of the 50 guides has been manually reviewed to strip stale absolute dollar figures from body text (e.g. "$400–$700+ NM"), since these go stale and duplicate what the Market column now does live. **This was NOT safely automatable** — two earlier automated attempts corrupted quotes and broke grammar. The process that worked, if you need to redo this for a newly-added album:
   - Classify every `$` mention into one of four buckets: **(1)** the guide's own stale current-value aside → delete cleanly; **(2)** a load-bearing comparative point (e.g. "commands a premium over X") → rewrite to preserve the *relative* claim without the number; **(3)** a price inside someone else's direct quotation → never touch; **(4)** a fixed historical fact/transaction (e.g. "sold for $255 in 2019 per [source]") → leave alone, it won't go stale.
   - Work one guide at a time, one line at a time, via `str_replace` — not a script. Verify with `grep -n '\$[0-9]' _content/X.md | grep -v "^[0-9]*:| "` before and after each guide, and re-run the audit after each one.
   - The only `$` mentions that should legitimately remain anywhere in `_content/` are bucket-3/4 exceptions (currently 5 across the whole corpus — a video title, a historical fact, three quotes).

## SEO / AI-Search Infrastructure (built this session)

- **`sitemap.xml`** — auto-generated at build time from `albums.json`, includes `<lastmod>` per album (see caveat below)
- **`robots.txt`** — explicitly welcomes AI crawlers as well as standard search bots
- **Structured data (JSON-LD)** on every album page: `Article` + `MusicAlbum` + `BreadcrumbList` combined via `@graph`; `WebSite` + `SearchAction` on the homepage; `Organization` (with `logo`) used consistently as author/publisher. **Deliberately did NOT use `Review`/`AggregateRating` schema** — the guides review pressings, not the album's music, and mismatched rating markup risks a Google penalty.
- **OG/Twitter tags** on every page, pointing at a proper 1200×630 `og-image.png` (not the small header logo stretched to fit)
- **Canonical URLs** everywhere; admin-only pages (`status.html`, changelogs) are `noindex`'d at both the meta-tag and Netlify-header level
- **Cover art resolution chain** (rebuilt twice this session — see "Known gotchas" below): mbid → Cover Art Archive (client-side, with a JS `onerror` fallback) → **server-side iTunes Search API lookup at build time** for anything without a working mbid. The client-side-only version was found to be unreliable (iTunes' CORS support is inconsistent in the wild), so the no-mbid case now resolves at build time and bakes a real `<img src>` directly into the HTML.
- **`Organization` logo, favicon, apple-touch-icon** — all derived from `branding/icon-source.png` (a proper square icon mark, replacing an earlier hand-drawn SVG placeholder)

### Explicitly considered and declined
- **FAQPage schema** — user's call; would've meant restructuring visible page content, declined to keep the page "clean"
- **llms.txt** — researched and recommended against. Current (mid-2026) evidence: major AI search crawlers largely ignore it for citation purposes; it's mainly used by coding agents (Cursor, Claude Code) reading software documentation sites, which doesn't apply here.
- **Full-text content search / a pressing comparison tool** — explicitly deferred by the user ("can wait"), not started

## Known Gotchas (things that bit us this session — worth knowing before you hit them again)

1. **GitHub API rate limits.** Unauthenticated requests cap at 60/hour/IP. An earlier version of `build.py` called `get_last_updated()` twice per album (once for sitemap `lastmod`, once for structured-data `dateModified`) — up to 100 calls in one build. Fixed by computing it once and passing it through, but `build_changelog()`/`build_status()` still make their own separate, larger calls per album and will compete for the same quota. If `lastmod`/`dateModified` come back mostly empty on a given build, this is why — not a bug, just quota exhaustion. Re-run later or authenticate.

2. **iTunes Search API CORS is unreliable.** Don't rely on a pure client-side `fetch()` to it for anything that needs to work reliably — resolve it server-side at build time instead (see `get_itunes_cover_url()` in `build.py`).

3. **PIL's `.ico` writer silently drops sizes** if you pre-shrink the source image yourself and pass `sizes=[...]` + `append_images=[...]`. Pass the *original full-resolution* image and let PIL do the resizing internally via the `sizes` parameter, or you'll get a single-size ICO despite requesting multiple.

4. **`albums.json`'s `year` field has inconsistent typing** (string vs. int across different entries) — cast to `int()` explicitly before doing arithmetic on it (bit the "Related guides" year-proximity sort).

5. **This sandbox (Claude's bash tool) cannot reach `itunes.apple.com`, `coverartarchive.org`, or most general internet domains** — only an allowlist (GitHub, PyPI, npm, etc.). Server-side code that calls these APIs can't be tested end-to-end from here; it'll only actually run successfully in Netlify's build environment, which has full internet access. Verify gotchas like this by reasoning about graceful-failure behavior, not by assuming a local test proves the happy path works.

## Research Protocol (the "7-lane deep pass," if resuming content research)

If picking research back up on any guide, this is the methodology that's been running:
1. **Exhaustive video coverage** — not just the first promising result; check multiple query variants and channels not yet used
2. **Steve Hoffman Forums** — genuinely underused relative to its density; search specifically, don't rely on it surfacing incidentally
3. **Print-magazine sweep** — Stereophile, TAS, Tracking Angle, AnalogPlanet
4. **Matrix/stamper-level detail** — LJC-depth specificity on "original pressing," not treating it as one monolithic entry
5. **Fact-audit pass** — re-verify existing claims against primary sources, not just new claims (this caught a fabricated Analogue Productions "Birth of the Cool" release and several wrong Discogs links this session)
6. **Regional pressing coverage** — beyond "the one obvious Japanese pressing"
7. **Recency check** — real time passes; genuinely new pressings turn up (found several this session: My Favorite Things' Rhino High Fidelity mono, Night Train's blue vinyl, A Love Supreme's 60th-anniversary Diamond Clear edition)

All 50 albums have been through at least one full pass. Diminishing returns are real — several albums (Bitches Brew, Sidewinder, Speak No Evil, Sunday at the Village Vanguard) came back clean on a pass with no changes needed, which is a legitimate outcome, not a failure to find something.
